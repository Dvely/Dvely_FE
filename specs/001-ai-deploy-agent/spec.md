## User Scenarios & Testing _(mandatory)_

### User Story 1 - 대화로 서비스 수정 반영 (Priority: P1)

비개발자가 자연어로 수정 요청을 보내고, 변경 결과를 미리보기에서 확인한 뒤 반영 여부를 결정한다.

**Why this priority**: 핵심 가치가 "대화만으로 수정"이므로, 이 흐름이 없으면 제품의 기본 사용 이유가 성립하지 않는다.

**Independent Test**: 기존 웹 프로젝트를 선택한 뒤 "헤더 문구 변경" 같은 요청을 보내고, 변경 미리보기와 반영 버튼까지 단독으로 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 사용자가 프로젝트 워크스페이스를 열어둔 상태, **When** 자연어 수정 요청을 전송하면, **Then** 시스템은 변경 요약과 검토 가능한 결과를 제시한다.
2. **Given** 변경 결과가 제시된 상태, **When** 사용자가 반영을 승인하면, **Then** 해당 요청은 반영 완료 상태와 함께 이력에 기록된다.

---

### User Story 2 - 배포 승인과 URL 획득 (Priority: P2)

사용자가 승인된 변경을 기준으로 배포를 실행하고, 배포 결과 URL과 상태를 확인한다.

**Why this priority**: 수정만 가능하고 배포가 불가능하면 실제 운영 가치가 떨어지며, README의 핵심 목표(운영 가능한 상태 도달)를 충족하지 못한다.

**Independent Test**: 반영 완료된 요청 1건을 기준으로 배포 승인 모달에서 실행 후, 배포 상태와 접근 가능한 URL 제공 여부를 확인한다.

**Acceptance Scenarios**:

1. **Given** 최근 승인된 변경이 존재하는 프로젝트, **When** 사용자가 배포를 승인하면, **Then** 시스템은 배포 진행 상태를 단계별로 보여준다.
2. **Given** 배포가 완료된 상태, **When** 사용자가 프로젝트 개요를 확인하면, **Then** 최신 배포 버전과 URL 정보를 확인할 수 있다.

---

### User Story 3 - 승인 정책과 작업 추적 관리 (Priority: P3)

사용자가 자동 승인 범위를 설정하고, 요청 단위로 변경·배포 결과 이력을 추적한다.

**Why this priority**: 운영 안정성과 신뢰를 위해 사용자가 자동화 강도를 제어하고, 문제 발생 시 이력을 되짚을 수 있어야 한다.

**Independent Test**: 설정 화면에서 자동 승인 정책을 변경한 뒤, 새 요청을 실행하여 정책 적용 여부와 요청별 이력 연결을 확인한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 설정 화면에 진입한 상태, **When** 자동 승인 대상을 변경하면, **Then** 이후 요청부터 변경된 정책이 적용된다.
2. **Given** 사용자가 특정 요청 이력을 조회하는 상태, **When** 상세 보기를 열면, **Then** 요청 내용, 반영 결과, 배포 결과가 연결된 형태로 표시된다.

---

### Edge Cases

- 배포 승인 직전 최신 반영 버전이 바뀐 경우, 사용자가 어떤 버전을 배포하는지 명확히 확인할 수 있어야 한다.
- 미리보기 생성이 실패한 경우, 사용자가 이해 가능한 원인과 다음 행동(재시도 또는 수정 요청 보완)을 안내받아야 한다.
- 외부 저장소 연동 권한이 만료된 경우, 사용자가 권한 재연결 없이는 반영/배포가 불가함을 즉시 인지해야 한다.
- 동일 요청이 연속 제출된 경우, 중복 실행 여부를 구분하여 불필요한 재배포를 방지해야 한다.

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: 시스템은 사용자가 새 프로젝트 생성, ZIP 업로드, 외부 저장소 불러오기 중 하나로 프로젝트를 시작할 수 있게 해야 한다.
- **FR-002**: 시스템은 자연어 요청을 받아 변경 대상, 변경 요약, 처리 상태를 사용자에게 제공해야 한다.
- **FR-003**: 시스템은 각 요청에 대해 미리보기 가능한 결과를 제공하고, 사용자가 반영 승인 여부를 선택할 수 있게 해야 한다.
- **FR-004**: 시스템은 승인 이벤트를 유형별(변경 반영, 배포, 도메인 연결)로 관리하고 요청 이력과 연결해야 한다.
- **FR-005**: 시스템은 승인된 변경을 기준으로 배포를 실행하고 배포 상태, 버전, 접근 URL을 사용자에게 제공해야 한다.
- **FR-006**: 시스템은 자동 승인 정책을 전체/수정/배포/도메인 단위로 설정·저장·적용해야 한다.
- **FR-007**: 시스템은 요청 단위로 "요청 내용 → 변경 결과 → 배포 결과"를 추적 가능한 기록으로 제공해야 한다.
- **FR-008**: 시스템은 미리보기 실패 또는 배포 실패 시 사용자가 이해 가능한 오류 설명과 우선 조치 제안을 제공해야 한다.
- **FR-009**: 시스템은 기본 UI/안내/AI 응답을 한국어 우선으로 제공해야 한다.
- **FR-010**: 시스템은 사용자 인증된 외부 저장소 계정에 대해 프로젝트 접근과 반영 작업 권한을 검증해야 한다.

### Key Entities _(include if feature involves data)_

- **Project**: 사용자가 작업하는 서비스 단위. 프로젝트 소스, 현재 URL, 배포 상태, 연결된 저장소 정보를 가진다.
- **User Request**: 사용자가 보낸 자연어 작업 단위. 요청 본문, 처리 상태, 변경 요약, 승인 여부를 가진다.
- **Approval Policy**: 자동 승인 범위를 정의하는 설정 단위. 전체/수정/배포/도메인별 정책 값을 가진다.
- **Deployment Record**: 배포 실행 결과 단위. 대상 버전, 실행 상태, 결과 URL, 실패 사유를 가진다.
- **Trace Record**: 요청-변경-배포를 연결하는 추적 단위. 감사 및 회귀 분석을 위한 링크를 제공한다.

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 첫 사용자 세션에서 80% 이상이 "프로젝트 시작 → 첫 수정 요청 → 미리보기 확인"까지 10분 이내 완료한다.
- **SC-002**: 승인된 변경의 95% 이상이 사용자 재시도 1회 이내에 배포 완료 상태에 도달한다.
- **SC-003**: 사용자의 주요 과업(수정 반영 또는 배포) 첫 시도 성공률이 85% 이상이다.
- **SC-004**: 배포 또는 미리보기 실패 케이스 중 90% 이상에서 사용자가 다음 행동을 선택할 수 있는 안내를 즉시 확인한다.
- **SC-005**: 요청 이력 조회 시 95% 이상에서 요청, 반영, 배포 결과의 연결 정보가 누락 없이 표시된다.

## Assumptions

- 초기 대상 사용자는 비개발자와 입문개발자이며, 복잡한 운영 용어보다 결과 중심 안내를 선호한다.
- MVP 범위에서는 정적 웹서비스 중심 시나리오를 우선 지원하며, 고급 인프라 오케스트레이션은 후속 단계로 둔다.
- 사용자는 프로젝트 접근에 필요한 외부 저장소 연동과 최소 권한 승인을 사전에 완료할 수 있다.
- 사용자는 반영/배포 전 승인 단계를 통해 자동화 위험을 통제하려는 의도가 있다.
